CHAPTER 4
DESIGN AND IMPLEMENTATION
4.0 Introduction
This chapter presents the full design and implementation of the Automated Energy Triage system. The project is a Hardware-based system designed to monitor residential electrical loads in real time, classify each load into a priority tier, track the prepaid energy budget continuously and autonomously shed non-essential loads before the budget reaches zero. Every component of the system runs on an ESP32 microcontroller with no dependency on cloud infrastructure or internet connectivity.
Chapter 2 identified a specific research gap: no affordable, autonomous, offline-capable energy triage device exists that targets the Zimbabwean residential prepaid electricity market. The system described in this chapter addresses that gap directly. The hardware layer uses SCT-013 current transformer sensors and an external relay module. The firmware layer implements four concurrent FreeRTOS tasks for sensing, classification, budget tracking and dashboard serving. The software layer provides a local Wi-Fi dashboard accessible from any browser on the same network.
This chapter covers the following content. Section 4.1 defines the project category and system objective. Section 4.2 describes the system architecture. Section 4.3 specifies functional and non-functional requirements, together with input, output, storage and security design. Section 4.4 presents the hardware design including circuit design, component selection and firmware architecture. Section 4.5 describes the implementation approach, tools, deliverables, milestones and schedule. Section 4.6 provides the chapter summary.
4.1 Project Category and System Objective
This project is a Hardware-based system. The design spans three integrated layers: a physical sensing and actuation layer built around the ESP32 microcontroller, SCT-013 CT sensors and relay modules; a firmware layer implementing real-time signal processing, load classification and triage decision logic; and a browser-accessible monitoring dashboard served locally from the microcontroller itself.
The system objective is to monitor residential electrical loads in real time using non-invasive current transformer sensors, classify each load into one of three priority tiers based on measured power consumption, continuously track the remaining ZESA prepaid energy budget and autonomously dispatch relay shedding commands to disconnect non-essential loads before the budget reaches zero, all without requiring internet connectivity, a mobile application or cloud infrastructure.
The system directly addresses the research gap identified in Chapter 2. No existing product in the Zimbabwean residential market combines affordable non-invasive sensing, tier-based autonomous triage and fully offline operation within a single embedded device costing under 50 USD.
4.2 System Architectural Design
4.2.1 Overall System Structure
The system uses a four-layer embedded IoT architecture. The layers are arranged vertically from physical sensing at the base to user monitoring at the top. Each layer has a single, clearly defined responsibility and communicates with adjacent layers through defined interfaces. This layered decomposition prevents coupling between sensing logic and decision logic, making each layer independently testable.
The four layers are: the Sensing and Control Layer (hardware), the Signal Processing Layer (firmware - Core 0), the Triage Decision Layer (firmware - Core 1) and the Monitoring Dashboard Layer (HTTP server on Core 1). The ESP32's dual-core architecture maps naturally onto this structure. Core 0 handles all time-critical sensing and classification. Core 1 handles budget tracking, triage decisions and web serving.

4.2.3 Justification of Design Choice
The layered embedded architecture is justified on three grounds. First, the ESP32 dual-core processor maps directly onto the layer separation: all time-critical sensing runs on Core 0 while budget logic and HTTP serving run on Core 1. This prevents dashboard activity from introducing jitter into the ADC sampling interval. Second, FreeRTOS task-based concurrency allows each module to run at an independently configured priority and stack size, giving explicit control over which work is preempted under load. Third, the physical separation of the relay module from the ESP32 sensing board is an architectural decision: relay coil switching transients can corrupt ADC readings if the relay is on the same board. The ribbon cable interface ensures the high voltage switching side never shares a ground plane with the CT sensor circuits.
The four-layer structure satisfies the non-functional requirements directly. The sensing layer runs continuously at the highest FreeRTOS priority, satisfying the 500 ms relay response target. The triage layer executes budget checks independently of HTTP requests, satisfying the offline operation requirement. The dashboard layer carries no dependency on external services, satisfying the zero internet connectivity requirement.

Security Control Specifications
The system operates on a local Wi-Fi access point created by the ESP32. It does not connect to the internet. This eliminates the largest surface for remote attack. The following security controls are implemented.
Table 4.7: Security Controls
Control
Implementation
Wi-Fi Password Protection
The ESP32 access point is configured with WPA2-PSK authentication. The default password is set at firmware compile time. No open AP mode is used.
Input Range Validation
All configuration inputs submitted through the dashboard are validated against defined acceptable ranges in the firmware HTTP request handler before being written to NVS. Out-of-range values are rejected with an HTTP 400 response and an error message.
Physical Isolation
The relay module is physically separate from the ESP32 sensing board. The relay control signals are opto-isolated. No mains voltage conductor is present on the ESP32 board or its immediate vicinity. This protects against electrical fault propagation to the microcontroller.
Firmware Integrity
The firmware is compiled and flashed as a binary image with the Arduino IDE. No over-the-air update mechanism is implemented, eliminating remote firmware modification as an attack vector.
Tier 1 Lock
A software constant defines the Tier 1 channel assignments. The triage engine contains a hard check that prevents any relay shed command from targeting a Tier 1 channel regardless of configuration or budget state.

4.4.3 Circuit Design
CT Sensor Interface Circuit
Each of the four CT sensor channels uses an identical interface circuit. The circuit performs three functions: it provides a burden resistor across the CT output terminals so the current output is converted to a proportional voltage, it establishes a 1.65V DC bias midpoint using a resistor voltage divider so the AC waveform sits centred in the ESP32's 0-3.3V ADC input range, and it provides capacitor filtering to reduce high-frequency noise on the signal line.

Triage Algorithm Pseudocode
The following pseudocode describes the triage decision logic executed by BudgetTriageTask when a threshold breach is detected. The pseudocode is language-independent and maps directly to the C++ implementation.

  PROCEDURE ExecuteTriage(budgetKwh, startKwh, critThreshPct)   
                                                                  
    budget_pct <- (budgetKwh / startKwh) * 100                  
                                                                  
    IF budget_pct <= critThreshPct THEN                          
                                                                  
      highest_tier <- 0                                          
      target_channel <- -1                                       
                                                                  
      FOR each channel ch IN [0 .. NUM_CHANNELS-1]              
        IF relayState[ch] = CLOSED AND tierAssign[ch] > 1 THEN  
          IF tierAssign[ch] > highest_tier THEN                  
            highest_tier  <- tierAssign[ch]                      
            target_channel <- ch                                 
          END IF                                                  
        END IF                                                   
      END FOR                                                    
                                                                  
      IF target_channel >= 0 THEN                                
        GPIO_Write(relayPin[target_channel], HIGH)               
        relayState[target_channel] <- OPEN                       
        LogTriageEvent(target_channel, budget_pct, millis())     
      END IF                                                      
                                                                  
    END IF                                                        
                                                                  
  END PROCEDURE

RMS Calculation Pseudocode
The following pseudocode describes the RMS current calculation and power derivation procedure executed by ClassificationTask for each ADC channel.

  PROCEDURE CalculateChannelPower(channel ch)                   
                                                                  
    emon.input <- adcRingBuffer[ch]                              
    rmsCurrentA[ch] <- emon.calcIrms(1480)                      
                                                                  
    IF rmsCurrentA[ch] < NOISE_FLOOR (0.1A) THEN                
      rmsCurrentA[ch] <- 0.0                                     
    END IF                                                        
                                                                  
    powerW[ch] <- rmsCurrentA[ch] * MAINS_VOLTAGE * POWER_FACTOR
                                                                  
    IF powerW[ch] <= tier1Boundary THEN                          
      tierAssign[ch] <- TIER_1                                   
    ELSE IF powerW[ch] <= tier2Boundary THEN                     
      tierAssign[ch] <- TIER_2                                   
    ELSE                                                         
      tierAssign[ch] <- TIER_3                                   
    END IF                                                        
                                                                  
  END PROCEDURE

User Interface Design
The system user interface is the local Wi-Fi dashboard served by the ESP32. The design principle is that a household member with no electrical engineering knowledge must be able to read the system status in under ten seconds. The dashboard is a single HTML page that auto-refreshes key data every two seconds without a full page reload.
The dashboard layout is organised into four visible zones. The Budget Zone at the top shows the remaining kWh value, the percentage of the original token remaining, and the estimated time before the budget reaches zero. This is the most important information and receives the most prominent position and largest text. The Circuit Status Zone shows four cards, one per monitored channel, each displaying the channel name, measured current in amperes, derived power in watts and the tier badge (Tier 1 in green, Tier 2 in amber, Tier 3 in red). The Relay Status Zone shows four toggle indicators showing which circuits are currently powered and which have been shed. The Triage Event Log at the bottom shows a timestamped list of the last twenty triage events.
The configuration form is accessible via a Settings link. It presents fields for starting kWh balance, critical threshold percentage and tier wattage boundaries. Input validation is performed in the browser before submission and again in the firmware on receipt.
4.5 Implementation
4.5.1 Implementation Overview
The system was implemented across four sequential phases, each building on the validated output of the previous phase. This approach ensures that each subsystem is confirmed correct in isolation before the integration overhead of combining multiple modules is introduced.
Programming Language and Tool Justification
The firmware is written in C++ using the Arduino framework. C++ is selected because it provides the low-level memory control required for ring buffer management and direct register access to the ESP32 ADC peripheral. The Arduino framework is selected because the ESP32 board support package and the EmonLib library are both mature, well-documented Arduino libraries. Writing the firmware in bare-metal C without the Arduino abstraction layer would require significantly more development time for ADC configuration, Wi-Fi initialisation and HTTP server setup without providing any performance improvement relevant to the 500ms triage response target.
The dashboard is implemented in HTML with inline JavaScript. No external JavaScript frameworks are used. A single HTML file is served from SPIFFS. This eliminates all network dependencies: the dashboard works with no internet connectivity and no CDN. The file size is kept below 15 KB to ensure fast loading over the ESP32 access point.
Python 3.11 with NumPy, Pandas and Matplotlib is used for evaluation data analysis. Python is selected for its conciseness for numerical analysis tasks and the maturity of the scientific computing libraries.

4.5.2 Implementation Phases
Phase 1: Hardware Assembly and CT Sensor Calibration
The CT sensor interface circuit was assembled on a solderless breadboard for initial testing. The burden resistor value of 68 Ohm was verified by measuring the peak output voltage at 30A and confirming the signal remained within the 0-3.3V ADC input range. The bias circuit was tuned to produce a stable 1.65V DC midpoint measured at the ADC input pin with the CT sensor disconnected.
Calibration was performed by clamping the SCT-013 sensor around a known resistive load conductor and comparing the ESP32's calculated RMS current against the Uni-T UT210E reference reading at ten current levels between 0.5A and 20A. A linear calibration coefficient was derived from the regression of ten repeated readings at each level and stored as the calibCal constant in the firmware.
Phase 2: Firmware Development
Firmware development followed a module-by-module approach. The Sampling Task was written and tested first, with the Arduino Serial Plotter used to visualize the live ADC waveform and confirm a clean, centred sinusoidal signal. The RMS calculation module was then integrated and the calibration coefficient applied. The tier classifier was added as a simple switch statement against the tier boundary constants. The budget tracker module was developed and tested with a mathematically simulated depletion scenario. The triage engine was added last, with the Tier 1 lock verified explicitly by attempting to trigger a shed command against a channel configured as Tier 1 and confirming the command was blocked.
Phase 3: Integration and Unit Testing
Each firmware module was tested independently before system integration. Integration testing confirmed that ADC samples flowed correctly to the RMS calculation module, that tier assignments updated within the 500ms classification cycle, that budget updates triggered triage commands at the correct threshold level and that relay state changes were reflected in the dashboard within two seconds of the switching event.
Phase 4: PCB Migration and Evaluation Preparation
The validated breadboard circuit was transferred to a general-purpose PCB using point-to-point wiring. The relay module was connected via the ribbon cable and mounted at a 15cm physical separation from the main board. The SPIFFS partition was flashed with the dashboard HTML file. The complete system was powered from the USB wall adapter and the dashboard was accessed from a mobile browser to confirm all display elements populated correctly before evaluation testing began.

4.6 Chapter Summary
This chapter presented the full design and implementation of the Automated Energy Triage system. The system is a hardware-based embedded IoT solution designed to monitor residential electrical loads in real time, classify each load into one of three priority tiers and autonomously disconnect non-essential loads before the ZESA prepaid energy budget reaches zero.
The chapter established a four-layer system architecture separating sensing, signal processing, triage decision logic and dashboard monitoring into distinct layers with defined interfaces. This architecture maps directly onto the ESP32 dual-core processor structure and the FreeRTOS multi-task firmware organisation. Functional requirements FR1 through FR10 specify every system capability with direct traceability to the research objectives from Chapter 1. Non-functional requirements define measurable performance, accuracy, cost and usability targets against which the system will be evaluated in Chapter 5.
Input, output, storage and security design specifications were presented. The hardware design described component selection and justification for every BOM item against the technical requirements and the 50 USD cost constraint. The CT sensor interface circuit was specified including burden resistor sizing and bias circuit design. Pin configurations were tabulated for all eight relevant GPIO connections. The firmware architecture was described through FreeRTOS task definitions, triage engine pseudocode and RMS calculation pseudocode.
System modelling tools used included a four-layer architecture diagram, a DFD Level 0 context diagram, a DFD Level 1 decomposition, a use case summary, a triage decision flowchart, a triage sequence diagram and a complete data dictionary. Each diagram maps directly to a firmware module or hardware component. Implementation was described across four sequential phases from hardware assembly through PCB migration and system integration testing. Ten project milestones with objective completion criteria and a fourteen-week project schedule were defined.
